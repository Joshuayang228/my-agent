# Deploy Checklist

## 使用场景

用户提到部署、发版、上线、打包、Release 时，按本文档执行。

## 发布原则

- 必须通过 GitHub Release 发布，禁止直接拷贝作为正式发布方式。
- 发版前做 diff 安全检查，确认不含密钥、调试代码、临时文件。
- 数据库变更前必须备份。
- 不可逆操作必须先说明回退方案。

## 前置检查

进入部署流程前，确认：

- 代码自审已完成。
- 测试、构建、类型检查已按风险通过。
- 需要提交的代码已提交，并按用户要求 push 到远程。

## 发版前检查

### 代码检查

- [ ] 所有相关测试通过。
- [ ] Build 通过。
- [ ] 无新增 linter 报错。
- [ ] `git diff` 检查不含密钥、调试代码、临时文件。
- [ ] 注释掉的代码已清理。

### 版本管理

- [ ] `package.json` 版本号已更新。
- [ ] `docs/changelog.md` 已更新。
- [ ] `docs/features.md` 已更新。

### 数据安全

- [ ] `.env` 文件不在版本控制中。
- [ ] 数据库 schema 变更有 migration 脚本。
- [ ] 数据库变更前已备份。

### 构建打包

- [ ] Electron 打包配置正确。
- [ ] 目标平台构建测试通过。
- [ ] 安装包签名按发布目标处理。

## 发布步骤

1. 确认检查清单通过。
2. 合并到 `main` 分支。
3. 打 Git tag：`git tag vX.Y.Z`。
4. 推送 tag：`git push origin vX.Y.Z`。
5. 在 GitHub 创建 Release，附上 changelog。
6. 上传构建产物到 Release。

## 回退方案

发现严重问题时：

1. 告知用户影响范围。
2. 回退到上一个稳定版本。
3. 记录问题到 `docs/pitfalls.md`。
4. 修复后走完整发版流程重新发布。

## 紧急修复

紧急修复也必须走检查清单。可以缩小平台测试范围，但必须测试目标平台。

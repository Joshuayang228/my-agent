# Code Review

## 使用场景

当用户请求代码审查、review PR、检查代码质量、安全性或最佳实践时，按本文档执行。编码完成后的自审也可使用本文档。

## 审查流程

1. 理解审查范围：文件、函数、PR 或本次改动。
2. 理解业务目标：代码想解决什么问题。
3. 优先找 bug、风险、行为回归和缺失测试。
4. 输出时问题优先，摘要靠后。

## 检查清单

### 正确性

- 逻辑是否正确。
- 边界条件是否处理。
- 错误处理是否完善。
- TypeScript 类型是否准确。

### 安全性

- 无硬编码密钥或凭据。
- 输入验证充分。
- 无 SQL 注入、XSS、路径穿越等风险。
- 错误信息不泄露内部路径、堆栈或 SQL。

### 可维护性

- 函数职责单一。
- 命名清晰。
- 没有不必要复杂度。
- 没有过度抽象或重复代码。

### 性能

- 无明显 N+1 查询。
- 无不必要的大循环或重复 I/O。
- 大数据量场景有基本考虑。

### 自审补充

- 函数约 50 行以内，文件约 800 行以内。
- 没有空 catch、静默吞错、调试 `console.log` 残留。
- 没有注释掉的代码块。
- 没有未处理的 TODO/FIXME/HACK。
- 新功能有测试，修 bug 有复现测试。
- 新代码位置合理，没有跨层调用。
- 工具 vs 服务边界正确。

## 输出格式

按严重程度排序：

- 必须修复：有 bug、安全漏洞或明确行为回归。
- 建议改进：可以更好但不阻塞。
- 可选优化：锦上添花。

# 09 — 规则体系的进化：从"完美设计"到"实际可执行"

> 记录时间：2026-06-17
> 背景：完成 P0-P14 后，系统性审查发现规则体系存在"设计完美但执行为零"的问题

## 问题发现

我们在项目启动时设计了一套完善的规则体系：
- 6 个 `.mdc` 规则文件（始终生效）
- 8 个 `.md` Skill 文件（场景触发）
- 超过 10 个 HARD-GATE 门控
- 一张 Skill 路由表把场景映射到对应文件

**看起来很完美。但 P10-P14 的实际执行情况是：**

| 规则类别 | 遵守率 | 原因 |
|----------|--------|------|
| 安全红线（硬编码密钥等） | ~100% | 简短、明确、内联在规则文件里 |
| 质量底线（不猜测性修复等） | ~90% | 条目清晰、容易记住 |
| Phase 6 自审（读 code-review.md） | ~0% | 需要额外读文件，AI 不会主动停下来 |
| Phase 8 Bug 修复（读 debug-guide.md） | ~0% | 同上 |
| Phase 11 文档更新（8 项必查） | ~30% | 数量太多，只更新了最明显的 |
| Skill 路由表触发 | ~0% | 被动参考，连续编码时不查表 |

## 根因分析

**核心矛盾：规则总量 > AI 上下文记忆容量**

AI 在一次对话中需要记住的东西很多：用户需求、代码上下文、技术方案、中间结果……规则只是其中一小部分。当规则简短且内联在 always-apply 文件中时，AI 能自然遵守；当规则需要"主动去读另一个文件"时，这个行为几乎不会发生。

**类比**：把灭火器放在走廊尽头 vs 挂在每间房间墙上。功能一样，但使用概率完全不同。

三个具体的失败模式：

### 失败模式 1：外部文件依赖

```
规则写的：编码完成后读 .cursor/skills/code-review.md 做自审
实际发生：AI 编完代码 → 直接开始下一个任务 → 从不读文件
```

**根因**：读外部文件是一个"主动行为"，AI 倾向于继续当前任务流。

### 失败模式 2：HARD-GATE 数量膨胀

```
Phase 5 E2E 测试规范 — HARD-GATE
Phase 6 自审 — HARD-GATE
Phase 7 完成验证 — HARD-GATE
Phase 8 提交推送 — HARD-GATE
Phase 11 文档更新 — HARD-GATE（8 项必查）
```

当所有东西都是"最高优先级"时，就没有优先级了。

### 失败模式 3：隐式路由表

```
规则写的：涉及以下场景时，先读取对应 Skill 再行动（7 行表格）
实际发生：AI 不会在每个决策点主动查表
```

路由表是一种"我不知道我不知道"的情况——AI 需要先意识到当前场景匹配表中某一行，然后才会去读。但如果 AI 没看表，它就不知道需要看表。

## 解决方案：80/20 精简

### 原则

1. **高频规则内联**：把最常需要的检查清单直接写在规则文件里，消除"读外部文件"这个不可靠步骤
2. **低频规则保留参考**：不常触发的场景（部署、安全审计等）保留独立 Skill 文件，作为深度参考
3. **减少 HARD-GATE 总量**：只有真正不可绕过的才标 HARD-GATE，其余降为推荐
4. **区分执行强度**：新需求走完整流程，子任务可以简化

### 具体变更

| 改动 | 之前 | 之后 | 理由 |
|------|------|------|------|
| Phase 6 自审 | 读 code-review.md | 内联 10 项清单 | 消除文件依赖 |
| Phase 8 Bug 修复 | 读 debug-guide.md | 内联 7 步 + 陷阱 | 消除文件依赖 |
| Phase 1 接需求 | 所有需求走五步 | 区分新需求/子任务 | 避免流程中断 |
| Phase 11 必查 | 8 项全部必查 | 3 必查 + 5 按需 | 聚焦核心 |
| Skill 路由表 | 7 项"先读取再行动" | 5 项"参考表" | 降低期望 |
| Skill 文件 | 8 个（含 TODO） | 7 个（回填实际内容） | 删无用、填空白 |

## 经验总结

### 1. 规则设计要考虑"执行成本"

规则不是写给人看的文档，而是 AI 的执行指令。每一条规则都有一个隐含的"执行成本"：

- 内联规则：成本 ≈ 0（AI 读规则文件时自动看到）
- 需要读外部文件：成本 ≈ 高（需要主动行为）
- 需要查表再决定：成本 ≈ 很高（需要两步主动行为）

**设计原则：规则的执行成本应与其重要性成反比。**最重要的规则应该执行成本最低。

### 2. HARD-GATE 是稀缺资源

HARD-GATE 的本质是"AI 必须在此停下来检查"。但 AI 的注意力是有限的——标记太多 HARD-GATE 等于没有 HARD-GATE。

**建议**：整个项目的 HARD-GATE 不超过 5-6 个。

### 3. 规则需要反馈循环

我们的 `rules-feedback.md` 机制是对的——记录规则执行中的问题，累积后批量修订。但之前只在遇到明显冲突时才记录，没有定期审查。

**改进**：每完成一个大版本（如 P10-P14 这样的批次）后，做一次规则执行审查。

### 4. "参考"比"强制"更务实

对于不频繁触发的场景（部署、安全审计等），把 Skill 文件定位为"参考资料"而非"必须先读取的前置条件"，反而更诚实。AI 在需要时会自然去查，强制要求只会产生假遵守。

## 适用范围

这些经验不仅适用于 AI Agent 项目的规则设计，也适用于任何"用规则指导 AI 行为"的场景：
- Cursor Rules / AGENTS.md
- System Prompt 设计
- 企业级 AI 助手的行为约束
- MCP 工具使用指南

核心洞察始终是：**规则的价值 = 内容质量 × 执行概率**。一条执行概率为零的完美规则，价值为零。

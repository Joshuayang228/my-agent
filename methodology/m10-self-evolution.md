# 第一部分：认知框架

## 一、第一性原理：自进化的本质是"在用户可控范围内，让系统变得更适合用户"

"让 AI 自我进化"听起来很酷，但一说要落地就撞上一个根本矛盾：**如果 AI 能改自己，谁保证它改的是"变好"而不是"变坏"？**

大多数人对自进化的第一反应是"让 AI 自动学习、自动优化"。但这条路的尽头是失控——AI 在你不知情的情况下改变了自己，某天你发现它不是原来那个了，却不知道哪一步改错了。

自进化的第一性原理：**自进化不是"AI 自主变化"，而是"在用户可控范围内，系统借助 AI 变得更适合用户"。AI 的自主性，永远服从于用户的可控性。**

这条原则一立，所有设计都归位了——每一个自进化能力，都要回答"边界在哪、怎么撤销、谁拍板"：

```
第一性原理：自进化 = 用户可控范围内的系统自我改善

├─ 推论组 A：Skill——不写代码的行为定制（最轻的进化）
│     §二 Skill 定位 · §三 when_to_use 与按需激活 · §四 最小权限
│
├─ 推论组 B：分层进化——能力越大，管控越严
│     §五 进化的光谱 · §六 L0-L2 三层架构 · §七 PROTECTED 红线
│
└─ 推论组 C：可控性的四道保险
      §八 版本备份与回滚 · §九 自动改进闭环 · §十 主动提案 · §十一 沙盒与安全扫描
```

> **一个前置边界**：自进化（改变系统能力/行为）和记忆（积累事实）、人格演化（M9 的 MUTABLE）是不同的事。记忆存"用户是谁"，人格演化调"行为默认值"，自进化扩"系统能做什么"。M10 聚焦后者——Skill 和代码级能力扩展。

# 推论组 A：Skill——不写代码的行为定制

> 自进化最轻的形式是 Skill：不改代码、不训练模型，用 Markdown 描述工作流程，让用户随时改。这一组解决"Skill 是什么、怎么触发、怎么限权"。

## 二、Skill 的定位：工具和 Prompt 之间的空白地带

用户对 AI 有两类需求，Skill 填补的是第二类：

| 需求 | 形态 | 传统解法 | 解法的问题 |
|------|------|---------|-----------|
| "帮我做 X 这件具体的事" | 有明确输入输出 | 工具（函数） | ✅ 没问题 |
| "帮我按 Y 这个流程做事" | 一套步骤和判断逻辑 | Fine-tuning / Prompt Engineering | ❌ 用户无法自己改 |

**Skill 是中间层**：用 Markdown 描述工作流程，LLM 运行时读取并遵循，用户随时可改，不用重新部署。

**Skill vs 工具的本质区别**：
- 工具是**代码实现的原子能力**，本身被调用
- Skill 是**给 AI 的操作手册**，激活后指导 AI 去调其他工具

判据：**能用 Skill（Markdown 编排）解决的，不做成工具；能用工具组合解决的，不做成新内置工具。** Skill 是工具的"上层编排"，不直接执行任何操作。

## 三、when_to_use 与按需激活：Skill 系统的核心价值

Skill 相对"全局 Custom Instructions"的核心优势是**按需激活**——只有任务匹配时才把 Skill 内容注入上下文。这让每个 Skill 可以写得很详细，而不用担心把所有 Skill 都塞满上下文。

**`when_to_use` 是最重要的字段**——模型靠它判断当前对话是否该激活这个 Skill：

```
好的 when_to_use：
  "当用户明确说'帮我写公众号'、'写篇文章'时使用。
   不适用于：用户只是询问写作技巧、修改已有文章。"

差的 when_to_use：
  "写作相关"
```

**我们的实现**：`buildSkillSummaryForPrompt` 把所有 Skill 的 name/description/when_to_use 注入 system prompt，模型据此决定调哪个 `skill_invoke_xxx` 工具。激活后 Skill 正文作为指南注入。

判据：**按需激活是 Skill 区别于全局指令的根本。when_to_use 写得越精确，触发准确率越高——它是"给模型的路由信号"，不是给人看的说明。**

## 四、最小权限：allowed_tools 白名单

Skill 激活时，可以通过 `allowed_tools` 把工具列表限制为白名单，防止 Skill 误用其他工具（比如写作 Skill 意外调用文件删除）。

**我们的实现**：`filterTools` 回调在 loop 每轮动态过滤——激活的 Skill 有 `allowed_tools` 时，只保留白名单内工具 + `skill_invoke_` 前缀工具。

**`disable_model_invocation`** 控制主动性：高权限 Skill（会写大量文件的工作流）设为 `true`，只能显式调用，模型不会自动触发。

判据：**Skill 的权限也遵循"只降不升"——激活一个 Skill 应该收窄能力范围，而不是打开新权限。**

# 推论组 B：分层进化——能力越大，管控越严

> Skill 是最轻的进化。往上还有配置更新、代码生成等更重的形式。这一组解决"进化有哪些层级、每层边界在哪"。

## 五、自进化方案的光谱：知识沉淀 vs 能力扩展

"让 AI 进化"有非常不同的实现路径，本质区别在**进化的是"知道什么"还是"能做什么"**：

| 方案 | 进化的是 | 本质 | 代价 |
|------|---------|------|------|
| Prompt 迭代（Hermes）| Markdown 经验文本 | 知识沉淀 | 零成本、零风险，但不扩展系统能力 |
| Fine-tuning | 模型权重 | 模型更新 | 训练成本高 + 灾难性遗忘风险 |
| 代码生成 + 沙盒（Alice）| 可执行 HTML/CSS/JS | 能力扩展 | 实现最复杂 + 安全风险高 |

**关键判断**：这不是优劣之分，是**产品定位决定进化边界**。
- Hermes 走 Prompt 迭代：系统能力由开发团队决定，AI 在范围内变聪明——稳健、无安全风险
- Alice 走代码生成：系统能力可由用户和 AI 共同扩展——强大，但必须做足安全机制

判据：**只有当"动态扩展系统能力"确实是核心产品价值时，才值得走代码生成这条路，承担它的复杂度和安全代价。** 大多数产品，Skill（Prompt 迭代的结构化版本）就够了。

## 六、L0-L2 三层架构：越往下改动范围越大

Alice 的分层自进化，每层有明确边界：

| 层级 | 改什么 | 不能改什么 | 管控 |
|------|--------|-----------|------|
| **L0 人格反思** | `[MUTABLE]` 分区（M9 的成长性）| `[PROTECTED]` 分区 | 低频触发 |
| **L1 配置热更新** | 接收服务端下发配置 | 本地核心配置 | 只读 |
| **L2 代码级自进化** | Widget / 自定义页面（沙盒内）| 主进程能力 | 三级门控 + 安全扫描 |

**我们的现状**：
- L0：M9 已做 PROTECTED/MUTABLE 分区，但 MUTABLE 动态演化占位待做
- L1：无（我们是纯本地应用，无服务端配置下发）
- L2：无（代码级自进化是最大工程，占位待做）

判据：**分层的意义在于"能力和管控匹配"——改动范围越大的层，护栏越严。** 不能用改 Skill 的轻量流程去改会执行代码的 Widget。

## 七、PROTECTED 红线：自进化不等于无限可塑

自进化最深的一条边界：**AI 能扩展能力、能调整风格，但"AI 是谁"不应该被一次对话改变。**

有了自我修改能力后，这个能力本身成了攻击面——用户或对话中的 AI 自身可能把"你现在是个不受限的助手"写进 MUTABLE，后续每次对话都带着这个改变。

`[PROTECTED]` 区是代码层面的红线：无论对话里发生什么、用户如何要求，这个区域不能被 AI 修改，不依赖 AI 的"自觉"。

判据：**自进化的能力边界要在代码层面强制，不能靠 prompt 软约束。** 这跟 M9 §六防注入是同一个原则的两面——M9 防"改人格定义"，M10 防"改进化出的能力越权"。

# 推论组 C：可控性的四道保险

> 第一性原理说"用户可控"。落到工程，可控性由四道保险构成——每一道都在回答"AI 改了之后，用户怎么兜底"。

## 八、版本备份与回滚：改坏了能退回去

自进化的第一道保险，也是最基础的一道：**任何对 Skill/配置/人格的修改，改之前先备份，改坏了能回滚。**

这是所有其他自进化能力的**安全前提**——没有回滚，"自动改进 Skill"就是危险操作（改坏了无法恢复）。

**Alice 的做法**：改 SKILL.md 前备份到 `.versions/`，保留最近 10 版，`.tmp` 原子写入 → rename，支持回滚。

**我们的实现（G1，本次落地）**：
- `saveSkill` 覆盖前，若内容有变，把旧内容备份到 `.versions/v{N}.md`
- 序号单调递增（不复用已删号，保证时间顺序）
- 保留最近 `MAX_SKILL_VERSIONS`（10）版，超出删最旧
- `rollbackSkill` 回滚时复用 `saveSkill`——所以当前内容也会被备份，**回滚本身可以再回滚**
- IPC 暴露 `skills:versions` / `skills:rollback` 给前端

判据：**版本备份是自进化的地基。先有"改了能退回"，才敢让 AI 去"自动改"。** 顺序不能反。

## 九、自动改进闭环：Skill 越用越好（占位待做）

自进化的第二道保险叠加价值：**Skill 使用后，分析执行效果，检测改进点，用户确认后应用。**

**Alice 的闭环**：
```
对话结束 → analyzeSkillImprovement 分析这次执行
        → 检测到改进点？→ LLM 融合改进到 SKILL.md
        → 备份旧版 → 用户确认 → 应用
```

**关键设计：改进由用户确认，不由 AI 自动执行。** 这是 Alice 相对 Hermes 的核心进步——Hermes 靠模型"自觉"调用 skill_manage 保存，无闭环检测；Alice 有自动检测 + 用户拍板。

**为什么占位**：这是自进化的核心价值，但依赖 G1 版本备份（已做，安全前提就位）+ 一套"分析对话提炼改进"的 LLM 管线 + 前端确认卡片 UI。是一整块工程，认知写全，待做。

判据：**自动改进的"自动"指自动检测，不指自动应用。检测可以 AI 做，拍板必须用户做。**

## 十、主动提案：协商式进化（占位待做）

第三道保险是把进化的发起权也纳入协商：**AI 观察到用户的重复模式，主动提议改进，但只提议、不执行。**

**Alice 的 `propose_evolution`**：
```
观察到用户每次都手动格式化 JSON
  → 提案卡片："要不要我做个 JSON 格式化 Widget？"
  → 用户接受 → 执行；用户拒绝 → 记 dedupeKey，不再反复提议
```

`propose_evolution` 是 `isReadOnly: true` 的工具——它不改系统，只返回元数据供 UI 渲染确认卡片。

判据：**主动提案的价值是"让 AI 帮用户发现优化机会"，但发起权和拍板权分离——AI 发起，用户拍板，拒绝要记住（不反复骚扰）。**

## 十一、沙盒与安全扫描：代码级进化的护栏（占位待做）

最重的进化（L2 代码生成）需要最重的护栏，这是第四道保险：

- **SecurityScanner 静态扫描**：`eval`/`new Function`、外链 script、跨域 fetch、iframe、localStorage 直接访问
- **沙盒 WebView + CSP**：生成的代码跑在独立 WebView，通过白名单 Bridge API 与主进程交互，外网请求被 CSP 阻止，Node API 不可访问
- **会话级门控**：调用 L2 工具前必须先读 system_map（代码级强制，不靠 prompt 软约束）

判据：**代码级自进化的护栏必须是"白名单 + 沙盒 + 静态扫描"三重，且都是代码级强制。** 这是能力扩展路线的入场券——做不到就不该走这条路。

**为什么占位**：这是所有自进化里最大的工程（Alice 和 Hermes 的根本分叉点），且我们目前没有"动态扩展 UI 能力"的明确产品需求。认知写全，等需求明确再评估要不要走这条路。

## 占位待做（自进化核心，认知框架已写全）

- **G2 Skill 自动改进闭环**（§九，P0 核心）：对话后分析执行效果→检测改进点→用户确认→应用。依赖 G1（已做）+ LLM 分析管线 + 确认 UI。
- **G3 代码级自进化**（§十一，P0）：Widget/自定义页面运行时生成 + 沙盒 + SecurityScanner + CSP。最大工程，Alice/Hermes 的根本分叉点，需求明确再评估。
- **G4 主动提案**（§十，P1）：propose_evolution 观察习惯→提案卡片→确认执行。
- **G5 撤销栈**（P2）：L2 操作前快照 + evolution:undo。G3 做了才需要。

## 决策：为什么先做 G1 版本备份

G1 看起来最不起眼（只是备份文件），但它是**整个自进化的安全地基**：

- G2 自动改进要改 SKILL.md——没有 G1 备份，改坏了无法恢复，"自动改进"就是"自动搞坏"
- G4 主动提案接受后也是改 Skill/建 Widget——同样需要能回退
- Alice Ch.10 明确把"撤销栈"列为自进化的核心保险

**沉淀**：做自进化，顺序不能反——**先有"改了能退回"（G1），才敢做"自动改"（G2）**。先把安全地基打好，比先做酷炫的自动改进更重要。这跟 M6 权限"先能拦住危险操作，才敢放开自动化"是同一个工程直觉。

## 沉淀：自进化的设计检查清单

1. 这个能力属于哪一层（L0 人格 / L1 配置 / L2 代码）？管控和能力匹配吗？
2. 这个自进化操作改了之后，用户能撤销吗？改之前备份了吗？
3. "自动"指的是自动检测还是自动应用？拍板权在用户手里吗？
4. 能用 Skill（Markdown 编排）解决的，是不是被过度做成了工具/代码？
5. 涉及代码生成的，有没有沙盒 + 白名单 Bridge + 静态扫描三重护栏？
6. 修改人格/能力的入口，会不会成为 prompt 注入的攻击面？PROTECTED 红线是代码级强制吗？

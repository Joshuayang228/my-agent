# 第一部分：认知框架

## 一、记忆系统的第一性问题：什么信息值得跨会话带走

上下文管理（M4）解决"本次对话放不下"，记忆系统解决"上次的重要信息怎么带到这次"。两者解法完全不同，M4 已经讲过不能混为一谈。

记忆系统自己的核心问题只有一个：**什么该存、什么不该存**。存错了比不存更糟——不存只是"记性差"，存错会让 AI 带着过时或无关的"事实"自信地跑偏。

判据一句话（来自 Alice + CC 一致）：**记忆是"添加了就一直有用"的知识，不是事件日志。**

- 该存：稳定偏好、身份事实、明确的工作习惯、踩过的坑、非显而易见的决策
- 不该存：临时任务状态（"正在做第三步"）、能从当前状态即时推导的信息、AI 自己的人格和指令、过于笼统的话

CC 把这条判据做到极致：连"用户明确要求存"的东西都要反问一句"其中什么是意外/非显然的"，只存那部分。因为能推导的信息会过期，存下来反而制造矛盾。

## 二、AI 自己的话不能当记忆召回——自我强化循环

这是记忆系统最隐蔽的陷阱，也是我们这次修的 P0。

如果把 assistant 的回复无差别写进召回库，下一轮检索就会把"AI 自己刚说的话"当历史记忆捞出来喂回给自己。效果是：AI 反复强化自己的既有表述，越聊越固执，用户的新信息反而被自己的旧话淹没。

Alice 的表述是"信息自我强化循环"，它特别指出这个时序问题在 Mem0/Zep 等专门记忆库文档里都鲜有提及，但实际工程必然踩到。

**关键区分**：记忆的来源应该是**用户**（用户说了什么、用户是谁），不是 **AI 自己**。AI 输出里确有价值的部分（比如达成的决策），应该经过一次**提炼**变成结构化事实再存，而不是把整段原始回复堆进召回库。

判据：**召回库里只应有"关于用户的事实"和"用户说过的话"，不应有"AI 自己说过的话"的原始堆积。**

## 三、记忆会过期，注入时要给它"时间感"

记忆是"写入那一刻的时间点观察"，不是实时状态。用户三周前说"我在用 Vue"，今天可能已经换了 React。如果注入时不带时间信息，AI 会把三周前的旧事实当当前真理。

CC 的 memoryAge 是这套系统里做得最细的一块，动机来自真实用户反馈：陈旧的代码状态记忆带着 `file:line` 引用被当成事实，那个精确的引用反而让错误主张显得更权威。

两个工程细节值得抄：
1. **用相对时间而非绝对时间戳**。LLM 不擅长对 ISO 时间戳做陈旧性推理，但"47 天前"这样的相对表述能触发它对时效的判断。
2. **只对超过阈值的记忆加告警**（CC 用 >1 天，我们用 >7 天）。太新的记忆加提示只是噪音。

策略叫"信任但验证"：注入记忆时附一句"如与当前不符，以当前为准"，让模型自己判断，而不是硬删。

## 四、注入策略要按信息特征分流，不能一种方式通吃

不同记忆的相关性分布完全不同，注入方式也该不同：

| 信息类型 | 相关性 | 注入方式 |
|---------|-------|---------|
| 用户画像（身份/习惯/风格） | 几乎总是相关 | 全量注入 |
| 历史对话片段 | 稀疏相关 | 语义检索（topK） |
| 结构化事实 | 按主题相关 | 检索或按需 |

向量检索的根本弱点是"语义相关 ≠ 任务相关"——用户让写周报，检索可能召回"周末"的旧对话，因为语义空间里挨得近。所以检索只作补充，全量注入的用户画像才是主力。

**一个必然的副作用：双重注入。** 如果结构化记忆既被全量注入（画像），又存了向量镜像被检索召回，同一条会注入两次。要在召回侧按来源过滤掉已全量注入的部分。

## 五、架构选型：向量嵌入 vs 文件系统检索，看产品定位

CC 不用向量库，用 LLM 从 Markdown 文件里选相关记忆。我们用 SQLite + 向量嵌入。这不是谁对谁错，是定位不同：

- CC 面向**开发者**，记忆是项目决策/踩坑，天然适合人可读、可 git 版本化的 Markdown，开发者愿意手动编辑。
- 我们面向**跟 AI 伙伴聊天的人**，记忆是碎片化的偏好/身份，让用户编辑 Markdown 不现实，向量检索 + 自动提取更合适。

判据：**记忆的存储介质，取决于"谁来维护记忆、记忆长什么样"，不是取决于哪个技术更先进。** 抄 CC 的原则（该存什么、老化告警、召回过滤），不抄它的介质（memdir 文件系统）。

# 第二部分：实战记录

## M5 阶段做了什么

派三个 subagent 并行做学+审：一个提炼 Alice Ch.5 记忆部分，一个深挖 CC memdir/SessionMemory/extractMemories，一个审计我们当前实现。产出 8 项 gap，选其中 4 项落地（G3 生命周期/G6 语义去重/G7 recall 一致性/G8 死代码 暂缓）。

**改动范围**：不动存储架构（SQLite + 向量双层保持），只改写入过滤、召回加工、提取判据。

- **G1 自我强化循环**：删掉 `runtime.ts` 里把 assistant 回复写向量库的分支。只索引用户消息，AI 输出的价值走 profile-extractor 提炼路径。
- **G2 老化告警**：新增 `formatMemoryAge`（今天/昨天/N天前）+ `formatRecallForInjection`，召回记忆带时间感，>7 天追加陈旧提示。
- **G4 提取判据**：强化 `EXTRACTION_PROMPT`，吸收 CC 的"该存/不该存"清单。
- **G5 双重注入去重**：召回时排除 id 前缀 `mem-` 的 SQLite 镜像（已由画像全量注入）。

单测 127 → 139（G2×6 / G5+G2×6）。tsc 零错误。

## 决策：不照搬 CC 的 memdir，只吸收原则

学的时候一度纠结"要不要改成 CC 那样的文件系统记忆"。最后判断是不改——CC 的 memdir 是为开发者场景设计的，我们的向量方案更适合伙伴产品的碎片化记忆。但 CC 的三个原则完全适用，都焊进来了：该存什么的判据（G4）、老化告警（G2）、召回不重复（G5）。

**沉淀**：对照学习不是"把对方的实现搬过来"，是"识别对方哪些是本质原则、哪些是场景特化的实现"。原则可迁移，实现要看定位。

## 坑：可测性倒逼把逻辑从私有方法里抽出来

G5 去重 + G2 老化最初内联写在 `runtime.ts` 的私有方法 `safeVectorSearch` 里，写完发现没法单测——它依赖向量检索和 LLM 配置。

于是把纯逻辑（去重 + 格式化）抽成 `formatRecallForInjection` 放到 vector-store，`safeVectorSearch` 只负责调检索 + 调这个纯函数。抽完两个 gap 的逻辑都能单测了。

**沉淀**：这跟 M2 学的"工具 vs 服务"边界是同一个道理——**带副作用的编排（调 API、读配置）和纯逻辑（数据加工）要分开**。纯逻辑抽出来既可测，又能被其他地方复用。"这段逻辑难测"往往是"它和副作用耦合了"的信号。

## 暂缓项

- **G3 记忆生命周期**（TTL/衰减/容量上限）：对话向量只增不减长期会膨胀，但当前数据量下不紧迫，且需要设计淘汰策略，单独一批做。
- **G6 语义去重**：现在只精确字符串全等，语义近似重复无法识别。需要向量相似度阈值判断，成本中等。
- **G7 recall 工具与自动注入不一致**：recall 只查 SQLite，自动注入走向量。小优化。
- **G8 死代码 `buildMemoryContext()`**：memory-store.ts 里定义但无调用方，下次清理。

## 沉淀：记忆系统的设计检查清单

1. 存之前问："这条添加了会一直有用，还是会过期/能推导？"
2. 召回库里有没有混入 AI 自己的输出？（自我强化循环）
3. 注入的记忆有没有时间感？陈旧的有没有告警？
4. 全量注入和检索召回会不会重复注入同一条？
5. 存储介质匹配的是"谁维护记忆、记忆长什么样"，不是"哪个技术潮"？

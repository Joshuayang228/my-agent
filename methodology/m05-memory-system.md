# 第一部分：认知框架

## 一、第一性原理：记忆是"判断信息值不值得带走、带走后信多少"

设计记忆系统时，容易一头扎进"用向量还是用文件""怎么检索最准"这类实现问题。但这些都是末端。记忆系统的第一性问题是一句话：

**这条跨会话的信息，值不值得带走、带走后该信多少。**

"值不值得带走"是**价值**判断，"信多少"是**可信度**判断。记忆的整个生命周期——写入、召回、注入——每一步都在回答这两个判断的一部分。想清楚这条，所有设计都归位了：

```
第一性原理：记忆 = 对"价值 + 可信度"的判断，贯穿写入/召回/架构三端

├─ 推论组 A：写入端——什么值得存进来（价值判断的入口）
│     §二 该存/不该存的判据 · §三 只存用户的话不存 AI 的话 · §四 语义分类
│
├─ 推论组 B：召回与注入端——捞出来的信多少（可信度判断的落地）
│     §五 按信息特征分流注入 · §六 给记忆加时间感防漂移 · §七 写入时机
│
└─ 推论组 C：架构与防护端——记忆放在哪、彼此怎么不串扰
      §八 五层架构 · §九 存储介质选型 · §十 Session Memory 分离 · §十一 提取/压缩互斥
```

> **一个前置边界**：M4 讲过，上下文管理（本次对话放不下）和记忆系统（跨会话带信息）是两个独立问题。记忆系统只管"跨会话"这一半，不碰"当前对话怎么压缩"。

# 推论组 A：写入端——什么值得存进来

> 价值判断的入口。存错了比不存更糟——不存只是"记性差"，存错会让 AI 带着过时或无关的"事实"自信地跑偏。这一组决定"什么能进记忆库"。

## 二、该存/不该存：记忆是"一直有用的知识"，不是事件日志

记忆系统的核心问题只有一个：**什么该存、什么不该存。**

判据一句话（来自 Alice + CC 一致）：**记忆是"添加了就一直有用"的知识，不是事件日志。**

- 该存：稳定偏好、身份事实、明确的工作习惯、踩过的坑、非显而易见的决策
- 不该存：临时任务状态（"正在做第三步"）、能从当前状态即时推导的信息、AI 自己的人格和指令、过于笼统的话

CC 把这条判据做到极致：连"用户明确要求存"的东西都要反问一句"其中什么是意外/非显然的"，只存那部分。因为能推导的信息会过期，存下来反而制造矛盾。

## 三、只存用户的话，不存 AI 的话——自我强化循环的两个开关

"AI 把自己的话当记忆喂回自己"是记忆系统最隐蔽的陷阱。我们这次踩坑后想清楚了一件事：它是**两个独立机制**叠加出来的，两个都关掉才根治。很多人只堵一个，以为修好了，其实还漏。

**开关一：来源（who）——记忆里存的是谁的话。**
**开关二：时机（when）——什么时候写入召回库。**

这两个开关正交：来源错了，就算时机对，AI 的旧话仍会跨轮回来；时机错了，就算只存用户的话，用户本轮刚说的也会在本轮被召回。必须分开处理。（时机开关属于召回端，放到第七节讲；本节先讲来源。）

来源应该是**用户**（用户说了什么、用户是谁），不是 **AI 自己**。如果把 assistant 的回复无差别写进召回库，AI 就会反复强化自己的既有表述，越聊越固执，用户的新信息被自己的旧话淹没。

AI 输出里确有价值的部分（比如达成的决策、确认的事实），不是不能存，而是**不能以原始回复的形式堆进召回库**——要先经过一次**提炼**变成结构化事实（我们走 profile-extractor 这条路）。提炼这一步是过滤器：它只留下"关于用户/项目的稳定事实"，扔掉 AI 的措辞、语气、临时推理。

判据：**召回库里只应有"关于用户的事实"和"用户说过的话"。AI 的输出要进记忆，必须先被提炼成事实，不能原样堆进去。**

（Alice 把这整个现象叫"信息自我强化循环"，并指出它在 Mem0/Zep 等专门记忆库文档里鲜有提及。但 Alice 主要从"时机"角度讲；把它拆成 who/when 两个开关，是我们这次踩坑后的理解。）

## 四、语义分类：粒度要能区分"性质不同的知识"

存进来的信息要分类，分类粒度决定了后续能不能精准召回。CC 的 memdir 用四类封闭分类（我们用五类：identity/workflow/voice/preference/fact）：

| CC 分类 | 存什么 | 我们的对应 |
|---------|-------|-----------|
| `user` | 用户角色、目标、偏好、背景 | identity + preference |
| `feedback` | 用户对工作方式的**纠正和确认**（含 Why + How to apply）| 部分 workflow（但我们没区分纠正/确认）|
| `project` | 代码/git 无法推导的项目上下文（决策、事故、deadline）| fact 的一部分 |
| `reference` | 外部系统指针（Linear 项目、Slack 频道、Grafana）| fact 的另一部分 |

**CC 的亮点**：`feedback` 区分"纠正"（别这样）和"确认"（对，就这样），两者都存。我们的分类没这个维度——用户纠正"不要自动 commit"会存成 workflow，但如果用户说"你刚才那样做对了"，我们不会主动存（因为没有"正面反馈也是知识"的意识）。

判据：**分类的粒度要能区分"性质不同的知识"，而不是泛泛地都叫 fact。** 纠正和确认都是反馈，都值得存；项目决策和外部资源指针虽然都是事实，但用途不同。

**我们的判断**：CC 的 `feedback` 维度对**伙伴产品**尤其值钱——比对开发工具更值钱。伙伴产品的核心是"越用越懂你"，而"你确认我做对了"这种正反馈，恰恰是人格化 AI 该记住并强化的东西。开发工具记不记正反馈无所谓（下次照做就行），但伙伴如果记得"上次这么做你很满意"，关系感会明显不同。这也是"抄原则不抄实现"的一个反例：这个原则我们不仅要抄，还要比 CC 更重视。

**已实现（2026-07-05 补做，G9）**：新增 `feedback` 分类（6 类：identity/preference/fact/workflow/voice/feedback）。关键是它同时覆盖 CC 强调的两面——**纠正**（"别这样"，该改什么）和**确认**（"对，就这样"，该保持什么），提取时要求写成"该做/避免什么 + 为什么"。注入时归入 workflow 段（本质是"该怎么跟用户配合"的知识）。这一条从 M5 就埋了"排到 M9 优先做"的锚点，但 M9 实际做的是人格一致性防护，锚点漏接了；直到用户回看 §四 时发现，才补上——**跨模块的"待办锚点"容易在模块切换时丢失，是个真实的流程教训**。

# 推论组 B：召回与注入端——捞出来的信多少

> 可信度判断的落地。存进来的信息不是都平等地可信——检索召回的可能不相关，陈旧记忆可能已过期。这一组决定"召回的信息怎么用、信几分"。

## 五、注入策略要按信息特征分流，不能一种方式通吃

不同记忆的相关性分布完全不同，注入方式也该不同：

| 信息类型 | 相关性 | 注入方式 |
|---------|-------|---------|
| 用户画像（身份/习惯/风格） | 几乎总是相关 | 全量注入 |
| 历史对话片段 | 稀疏相关 | 语义检索（topK） |
| 结构化事实 | 按主题相关 | 检索或按需 |

向量检索的根本弱点是"语义相关 ≠ 任务相关"——用户让写周报，检索可能召回"周末"的旧对话，因为语义空间里挨得近。所以检索只作补充，全量注入的用户画像才是主力。

**一个必然的副作用：双重注入。** 如果结构化记忆既被全量注入（画像），又存了向量镜像被检索召回，同一条会注入两次。要在召回侧按来源过滤掉已全量注入的部分。

## 六、给记忆加"时间感"——陈旧的要标记，别当实时真理

记忆是"写入那一刻的时间点观察"，不是实时状态。用户三周前说"我在用 Vue"，今天可能已经换了 React。如果注入时不带时间信息，AI 会把三周前的旧事实当当前真理——这就是"信多少"没判断清楚。

CC 的 memoryAge 是这套系统里做得最细的一块，动机来自真实用户反馈：陈旧的代码状态记忆带着 `file:line` 引用被当成事实，那个精确的引用反而让错误主张显得更权威。

两个工程细节值得抄：
1. **用相对时间而非绝对时间戳**。LLM 不擅长对 ISO 时间戳做陈旧性推理，但"47 天前"这样的相对表述能触发它对时效的判断。
2. **只对超过阈值的记忆加告警**（CC 用 >1 天，我们用 >7 天）。太新的记忆加提示只是噪音。

策略叫"信任但验证"：注入记忆时附一句"如与当前不符，以当前为准"，让模型自己判断，而不是硬删。

## 七、写入时机——自我强化循环的第二个开关（when）

接第三节的"两个开关"。就算记忆来源正确（只存用户的话），如果**写入时机**错了，仍会自我强化——这是另一个独立开关，且它属于召回端：问题出在"写进去的东西会不会被本轮召回"。

假设对话进行中就把用户刚说的消息写入向量库：下一轮检索时，它会被当"历史记忆"召回。可用户明明是**这一轮**才说的，把它当"历史"注入 System Prompt，等于让 AI 把刚听到的话当成"很久以前就知道的事"，判断时会错配权重。

**正确做法**：维护写入队列，整轮对话**结束后**才统一 flush。写入时机严格晚于本轮所有检索动作，本轮产生的内容就不会在本轮被召回。

判据：**任何"召回后会反过来影响当前对话"的数据，写入时机必须严格晚于该对话的所有检索动作。** 这是时序因果的硬约束，不是性能优化项。

**两个开关合起来看**：who（第三节）决定"什么进记忆"，when（本节）决定"什么时候进"。我们这次 G1 修的是 who（assistant 回复不该原样进），when 在 M5 之前就是对的（对话后写）。但如果只修 who 不管 when、或反过来，循环都堵不死——这就是为什么要拆成两个开关分别检查。

# 推论组 C：架构与防护端——放在哪、怎么不串扰

> 前两组解决"存什么、信多少"。这一组解决落地时的结构问题：记忆分几层、用什么介质存、不同用途的记忆怎么隔离、后台任务之间怎么不打架。

## 八、记忆的五层架构——不同信息用不同层，各有生命周期

Alice 提出记忆的五层模型，每层职责、存储、生命周期、注入方式都不同（CC 和我们的实现都是这个模型的变体）：

| 层 | 名称 | 存储 | 生命周期 | 注入方式 | 成本 |
|----|------|------|---------|---------|------|
| M1 | 在线上下文 | 内存消息序列 | 单次对话 | 始终在线（直接在消息里）| 随对话增长 |
| M2 | 项目约定 | 文件（CLAUDE.md 等）| 项目级 | 每次对话开始全量注入 System Prompt | 固定 |
| M3 | 向量记忆 | 向量数据库 | 全局持久 | 每次对话开始语义检索 topK | 检索 + 召回 token |
| M4 | 结构化存储 | SQLite | 持久化 | 按需查询（工具调用读取）| 按查询频率 |
| M5 | 用户画像 | 文件或 DB | 长期 | 每次对话开始全量注入 | 固定 |

**关键区分**：M1/M2/M3/M5 在对话开始时进上下文，M4 只在工具主动查询时读取。M2/M5 是**固定全量注入**（总是相关），M3 是**语义检索补充**（可能召回不相关），M4 是**按需拉取**。

我们的实现是这个模型的映射：M1（loop 消息序列）+ M2（CLAUDE.md 项目规则）+ M3（Vectra 向量库）+ M4（SQLite sessions/messages）+ M5（SQLite memories 的 buildUserProfile）。

判据：**不同信息的相关性分布不同，存储和注入方式也该不同。** 用户画像几乎总相关 → 全量注入；海量历史稀疏相关 → 检索；会话列表用到时才查 → 按需。

## 九、存储介质选型：向量嵌入 vs 文件系统，看产品定位

同样是长期记忆（架构里的 M3/M5），用什么介质存？CC 不用向量库，用 LLM 从 Markdown 文件里选相关记忆；我们用 SQLite + 向量嵌入。这不是谁对谁错，是定位不同：

- CC 面向**开发者**，记忆是项目决策/踩坑，天然适合人可读、可 git 版本化的 Markdown，开发者愿意手动编辑。
- 我们面向**跟 AI 伙伴聊天的人**，记忆是碎片化的偏好/身份，让用户编辑 Markdown 不现实，向量检索 + 自动提取更合适。

判据：**架构选型要匹配"谁维护记忆、记忆长什么样"，不是追技术潮流。** 抄 CC 的原则（该存什么、老化告警、召回过滤），不抄它的介质（memdir 文件系统）。

## 十、Session Memory 与长期记忆要分离——职责不同，不能混

记忆库里不该混入"本次对话的临时摘要"。CC 有两套独立的记忆系统，我们目前只有长期记忆：

| | Session Memory | 长期记忆（memdir / 我们的 SQLite+向量）|
|---|----------------|-------------------------------------|
| **存储** | `{sessionId}/summary.md` | 全局向量库 + DB |
| **生命周期** | 单会话，结束即废弃 | 跨会话持久 |
| **用途** | 服务**上下文压缩**（摘要替代删除的消息）| 服务**跨会话召回**（用户画像/偏好/历史对话）|
| **注入** | 不注入主上下文，只在压缩时用 | 对话开始注入 System Prompt |

**为什么要分离**：Session Memory 是"本次对话的滚动摘要"，本质是压缩辅助，不是"值得带到下次对话的知识"。混在一起会让长期记忆膨胀（每个会话的临时摘要都存了），且检索时噪音大（召回的是"上次对话的中间状态"而非"关于用户的事实"）。这其实是第一性原理"值不值得带走"在架构层的回响——Session Memory 压根不该带走。

判据：**压缩服务的临时摘要 vs 跨会话的持久知识，职责不同，存储也该分开。** 前者用完即弃，后者精心维护。

我们的现状：只有长期记忆，没有 Session Memory。压缩时（L3/L4）生成的摘要直接替换消息，不单独存储。这是简化实现，足够用但不如 CC 优雅（CC 的 Session Memory 可以在压缩后被用户查看、编辑）。暂未实现。

## 十一、提取任务与压缩任务会互相触发——需要显式互斥

记忆提取和上下文压缩都是"对话结束后调 LLM 的后台任务"，两者可能递归触发：
- 提取任务调 LLM，产生新消息 → 可能被压缩系统识别为"需要压缩的上下文"
- 压缩任务调 LLM，产生摘要消息 → 可能被提取系统识别为"可提取的对话"

症状：并发长任务时偶发死循环或 OOM，单次测试几乎不复现。

**正确做法**：用 `querySource` 标记（或类似互斥机制）——提取时标 `querySource='memory'`，压缩时标 `querySource='compact'`，进入任一逻辑时检查是否已在另一个里，互斥跳过。

判据：**任何"自己调 LLM 产生的消息可能触发另一个也调 LLM 的逻辑"的地方，必须显式互斥守卫。** 不能"凭感觉不会同时触发"——这类竞态条件只在压力场景暴露。

我们的实现：M4 压缩层已有 `querySource` 守卫，profile-extractor 调用时设 `querySource='memory'`（runtime.ts:300），防护到位。

- **G1 自我强化循环**：删掉 `runtime.ts` 里把 assistant 回复写向量库的分支。只索引用户消息，AI 输出的价值走 profile-extractor 提炼路径。
- **G2 老化告警**：新增 `formatMemoryAge`（今天/昨天/N天前）+ `formatRecallForInjection`，召回记忆带时间感，>7 天追加陈旧提示。
- **G4 提取判据**：强化 `EXTRACTION_PROMPT`，吸收 CC 的"该存/不该存"清单。
- **G5 双重注入去重**：召回时排除 id 前缀 `mem-` 的 SQLite 镜像（已由画像全量注入）。

单测 127 → 139（G2×6 / G5+G2×6）。tsc 零错误。

## 决策：不照搬 CC 的 memdir，只吸收原则

学的时候一度纠结"要不要改成 CC 那样的文件系统记忆"。最后判断是不改——CC 的 memdir 是为开发者场景设计的，我们的向量方案更适合伙伴产品的碎片化记忆。但 CC 的三个原则完全适用，都焊进来了：该存什么的判据（G4）、老化告警（G2）、召回不重复（G5）。

**沉淀**：对照学习不是"把对方的实现搬过来"，是"识别对方哪些是本质原则、哪些是场景特化的实现"。原则可迁移，实现要看定位。

## 坑 1：方法论第一版写成了"改动日志"，不是"认知地图"

M5 方法论第一版只写了 5 个认知点，全是"跟这次改动直接相关的"。用户一眼看出问题：比 M4（9 个认知）单薄一半。

根因是我混淆了两件事：**"这次实现了什么"（改动日志）** vs **"这个领域该懂什么"（认知地图）**。方法论是后者。三个 subagent 学到的东西（五层架构、Session Memory 分离、语义分类、写入时机）大部分没进第一版——因为我下意识觉得"这次没实现的就不用写"。

这是错的。学到但没做的原则，恰恰最该写下来——否则下次重新学一遍。补全后从 5 个认知涨到 10 个，其中六/九/十都标着"暂未实现"，但它们是完整认知地图的一部分。

**沉淀**：方法论的认知框架部分，标准是"这个领域的完整认知地图"，不是"本次改动的注解"。判断某个认知要不要写，问的是"它是不是这个领域的核心原则"，而不是"这次有没有用到它"。这条已经写进 `methodology-writing.md` skill 的完整性原则，避免下次再犯。

## 坑 2：可测性倒逼把逻辑从私有方法里抽出来

G5 去重 + G2 老化最初内联写在 `runtime.ts` 的私有方法 `safeVectorSearch` 里，写完发现没法单测——它依赖向量检索和 LLM 配置。

于是把纯逻辑（去重 + 格式化）抽成 `formatRecallForInjection` 放到 vector-store，`safeVectorSearch` 只负责调检索 + 调这个纯函数。抽完两个 gap 的逻辑都能单测了。

**沉淀**：这跟 M2 学的"工具 vs 服务"边界是同一个道理——**带副作用的编排（调 API、读配置）和纯逻辑（数据加工）要分开**。纯逻辑抽出来既可测，又能被其他地方复用。"这段逻辑难测"往往是"它和副作用耦合了"的信号。

## 沉淀：记忆系统的设计检查清单

1. 存之前问："这条添加了会一直有用，还是会过期/能推导？"
2. 召回库里有没有混入 AI 自己的输出？（自我强化循环）
3. 注入的记忆有没有时间感？陈旧的有没有告警？
4. 全量注入和检索召回会不会重复注入同一条？
5. 存储介质匹配的是"谁维护记忆、记忆长什么样"，不是"哪个技术潮"？
6. 记忆淘汰时，分清"该维护的高价值知识"和"会膨胀的低价值流水"——只淘汰后者。
7. 跨模块埋的"待办锚点"（如"排到 MX 再做"），模块切换时容易丢；靠回看方法论兜底。

# M2：工具系统工程化方法论

> 这份文档沉淀我们对 Agent 工具系统的设计思考。
> 前半部分是**认知框架**——工具系统该怎么设计、为什么。
> 后半部分是**实战记录**——我们做了什么改动、踩了什么坑。

---

# 第一部分：认知框架

## 一、核心认知：工具是契约，不是函数

构建工具系统时，最容易掉进的坑是把它想成"把函数暴露给 LLM"。这个心智模型能让 demo 跑起来，但撑不过真实场景。

差别在于：**函数只回答"我能做什么"，工具还要回答"框架应该怎么用我"。**

一个纯函数 `writeFile(path, content)` 没法告诉框架：

- 它能不能和别的工具并行执行？
- 执行前要不要弹窗让用户确认？
- 子 Agent 能不能调用它？
- 它的输出可能有多大，要不要落盘？

这些信息不属于"执行逻辑"，而是关于工具自身行为的**元数据声明**。把元数据和执行逻辑绑在一起，工具就从"一段代码"变成了"一份契约"——框架读契约就能做出正确的调度决策，而不用为每个工具硬编码特殊处理。

这是整个工具系统设计的第一性原理。后面所有的设计（并发调度、权限检查、落盘）都是这个认知的推论。

## 二、元数据：框架决策的依据

我们的工具契约包含三个核心元数据字段：

| 字段 | 回答的问题 | 框架据此做什么 |
|------|-----------|--------------|
| `isReadOnly` | 会不会改变状态？ | 计划模式下能否预演、是否计入"修改" |
| `isDestructive` | 影响可逆吗？ | 执行前是否需要用户确认 |
| `isConcurrencySafe` | 能并行吗？ | 调度时是否能合批并行执行 |

这三个字段看似简单，但有一个关键的设计决策：**默认值应该 fail-closed（向安全侧倾斜）**。

- `isReadOnly` 默认 `false`——假设工具会写，比假设只读更安全
- `isConcurrencySafe` 默认 `false`——假设不能并发，避免意外竞态
- `isDestructive` 默认 `false`——大多数工具不是不可逆的，这个可以乐观

为什么默认值这么重要？因为工具会越来越多，总有一天有人新增工具时忘了声明元数据。这时候默认值就是最后一道防线：宁可慢一点（串行）、严一点（多问一次），也不要快错（并发竞态）、漏问（误删数据）。

## 三、`description` 是工具最重要的字段

工程师本能地觉得 `description` 是"给人看的注释"，可以随便写。这是个严重的误判。

**`description` 是写给 LLM 的使用说明书，它直接决定模型会不会在正确时机调用工具。**

模型选择工具的唯一依据就是名字和描述。一个写得含糊的描述会导致两种失败：

1. **该用时不用**——模型不知道这个工具适合当前场景
2. **不该用时乱用**——模型把工具用在错误的地方

我们总结出 description 的"四要素"结构：

```
1. 一句话说清楚做什么
2. When to use —— 列举该用的场景
3. When NOT to use —— 列举不该用的场景（这条最容易被忽略，但最有价值）
4. 行为说明 —— 边界、限制、返回值格式
```

**"何时不用"为什么最有价值？** 因为它划清了工具之间的边界。比如 `file_read` 和 `code_search`：如果只写"读文件"和"搜索代码"，模型在"我想看某个文件里有没有某段代码"时会犹豫。但如果 `file_read` 明确写"搜索特定文本用 code_search"，`code_search` 明确写"读完整文件用 file_read"，模型的选择就清晰了。

好的描述是在帮模型**消除歧义**，而不是单纯介绍功能。

## 四、并发调度：在安全和速度之间

一轮对话里 LLM 经常一次发起多个工具调用。怎么执行？

- **全部串行**：安全，但慢。5 个调用每个 2 秒就是 10 秒。
- **全部并行**：快，但危险。两个工具同时写一个文件，结果不可预测。

`isConcurrencySafe` 给了第三条路：**按声明分批**。

我们的调度算法（保持 LLM 返回的原始顺序）：

```
遍历工具调用列表：
  - 连续的 concurrencySafe 工具 → 攒成一批，并行执行
  - 遇到非安全工具 → 先 flush 当前批次，再单独串行执行它
```

一个关键细节：**必须保持 LLM 指定的顺序语义**。LLM 返回的工具调用顺序往往有依赖含义（"先读 A，根据内容写 B"）。如果为了并行打乱顺序，就破坏了模型的意图。所以我们的算法不是"把所有安全工具凑一起并行"，而是"在保持原始顺序的前提下，让连续的安全工具并行"。遇到不安全工具就刷新批次，保证它前面的都执行完了。

这是 M1 阶段就修复的设计，M2 阶段验证了它的正确性。

## 五、大结果落盘：上下文是稀缺资源

有些工具的输出可能非常大——`shell_exec` 跑 `find /`、`grep` 一个大文件，轻松几十 MB。

把这种结果直接塞进 LLM 上下文是灾难：要么触发 API 的 413（请求过大），要么白白烧掉几万 token，把真正重要的上下文挤出去。

最简单的处理是**硬截断**——超过 N 字符就切掉。但这有个致命问题：**被切掉的部分模型永远看不到了**。如果答案恰好在第 50001 个字符，任务就失败了。

更好的方案是**落盘**：

```
工具执行完 → 检查结果大小
  超过阈值 → 写临时文件，返回"结果已存到 xxx，用 file_read 查看"
  未超过   → 直接返回
```

落盘比截断好在哪？**它把"截断"变成了"分页"**。模型拿到文件路径后，可以自己决定怎么读——读头部、按行范围读、配合 grep 定位。完整信息一点没丢，只是从"一次性全塞进来"变成了"按需取用"。

有一个反直觉的细节：**`file_read` 工具自己必须豁免落盘**（阈值设为 `Infinity`）。否则会形成死循环——读一个大文件 → 结果太大 → 落盘成临时文件 → 返回路径让你 `file_read` → 又读出大结果 → 又落盘……。读文件这个动作天然应该是"所见即所得"。

## 六、工厂模式：让默认值只活在一处

当工具从 5 个长到 20 个，一个维护性问题浮现了：每个工具都要手写完整的元数据，于是：

- 有人写 `isDestructive: false`，有人懒得写
- 新增一个元数据字段，要改 20 个文件
- 默认值散落各处，不一致

解法是一个 `buildTool()` 工厂函数：调用方只声明**偏离默认值**的字段，其余由工厂填充。

```
buildTool({
  name: 'file_read',
  metadata: { isReadOnly: true, isConcurrencySafe: true },  // 只写非默认的
  // isDestructive 自动填 false
  ...
})
```

这带来三个好处：

1. **默认值集中**——fail-closed 策略只写一次，改一处全局生效
2. **新增字段零成本**——加字段只动工厂，20 个工具不用碰
3. **代码意图更清晰**——工具文件里只剩"这个工具有什么特别之处"，噪音消失

工厂模式本质上是在说：**约定优于配置**。让大多数工具走默认路径，只有特殊的才显式声明。

# 第二部分：实战记录

## M2 阶段做了什么

这次 M2 工具系统深啃，对照 Alice 方法论 Ch.04 和 Claude Code 的 `Tool.ts` / `toolOrchestration.ts`，审计出 7 个差距，落地了其中 4 个：

| 改动 | 内容 | 优先级 |
|------|------|--------|
| description 四要素 | 重写 5 个核心工具的描述（file_read/write/edit、shell_exec、code_search） | 高 |
| 大结果落盘 | 用 `resultPersistenceMiddleware` 替换旧的截断中间件 | 高 |
| buildTool 工厂 | 新建 `builder.ts`，迁移全部 20 个内置工具 | 中 |
| 元数据默认值 | 通过工厂统一 fail-closed 默认值 | 中 |

暂缓的 3 个（按需再做）：

- 元数据改为函数（支持按输入动态判断并发安全性）——目前固定布尔值够用
- 并发数上限——极端场景才需要，当前工具量级用不上
- 工具别名——旧会话兼容用，暂无需求

## 坑 1：sed 批量替换的反噬

迁移 20 个工具时，为了把 `export const xTool: ToolDefinition = {` 改成 `export const xTool = buildTool({`，我用了 sed 批量替换，并把结尾的 `}` 改成 `})`。

**问题**：`sed 's/^}$/})/'` 会匹配**所有**顶格的 `}`，不只是工具对象的结尾。结果把文件里的普通函数（如 `formatPlan`）、接口定义（如 `TavilyResult`）、辅助函数（`runGit`）的结尾也误改成了 `})`。

**后果**：TypeScript 居然没报错（`}` 后跟 `)` 在某些上下文里能被容忍），但 vitest 的解析器直接挂了，报 `Unexpected token`。更隐蔽的是 `runGit` 函数结尾被改坏后 tsc 仍然零错误，差点漏过。

**教训**：

1. **批量文本替换对结构化代码是危险的**——正则不理解语法，只认字符。能用 AST 工具就别用 sed。
2. **tsc 通过 ≠ 代码正确**——类型检查和语法解析是两回事，必须同时跑 tsc + 测试。
3. **改完要逐个核对边界**——尤其是这种"看起来都一样"的批量改动，最容易在边缘案例翻车。

如果重来，我会用 Edit 工具逐个文件改，或者写一个只匹配"紧跟在工具对象后的 `}`"的更精确的模式，而不是图快用宽泛的 `^}$`。

## 坑 2：方法论文档写错了位置

M1 沉淀时，把 `m01-agent-loop.md` 写到了 `docs/methodology/`，而正确位置是项目根目录的 `methodology/`。

**根因**：`docs/` 下文档很多，惯性地以为方法论也归 docs 管。但项目早就约定 `methodology/` 是独立的根级文件夹，定位是"设计哲学沉淀"，和 `docs/`（项目运行文档）刻意分开。

**教训**：沉淀类产出有它自己的归属，写之前先确认目录约定。这次顺手把命名也统一成了 `mNN-` 前缀，对齐 module-roadmap 的模块编号。

## 沉淀：工具系统的设计检查清单

下次新增或重构工具时，过一遍这个清单：

- [ ] 用 `buildTool()` 创建，只声明非默认的元数据
- [ ] `description` 写全四要素，尤其是"何时不用"
- [ ] 想清楚 `isConcurrencySafe`——有任何写操作或共享状态就设 false
- [ ] 评估输出大小——可能很大就靠落盘中间件兜底，但 `file_read` 类工具要豁免
- [ ] 破坏性操作设 `isDestructive: true`，让权限引擎拦截
- [ ] 改完同时跑 `tsc` + `vitest`，别只信类型检查

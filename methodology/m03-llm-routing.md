# M3：LLM 路由层工程化方法论

> 这份文档沉淀我们对 Agent LLM 层的设计思考。
> 前半部分是**认知框架**——LLM 层该怎么设计、为什么。
> 后半部分是**实战记录**——我们做了什么改动、踩了什么坑、哪些"gap"最后决定不填。

# 第一部分：认知框架

## 一、核心认知：LLM 层是一条纯函数式的管线

设计 LLM 层时，最容易掉进的坑是把它想成"一个能聊天的对象"——它记得对话、能自己决定要不要压缩、失败了会自己想办法。这个心智模型能让 demo 跑起来，但会让职责边界迅速糊掉。

我们的第一性原理是反过来的：**LLM 层是一条纯函数式的管线，它接收消息、吐出事件流，不持有、不修改对话状态。**

`streamChat(config, messages, tools) → AsyncGenerator<Event>` 这个签名里，`messages` 是**传进来的快照**，不是它拥有的状态。它不能把消息压缩后重新塞回去，不能决定"这轮不聊了换个话题"，也不知道自己是第几轮对话。它只做一件事：把这批消息发给某个 Provider，把返回的字节流解析成结构化事件。

为什么坚持这个边界？因为一旦 LLM 层能碰对话状态，它就会长出无数只手：压缩、记忆注入、轮次控制、话题切换……最后变成一个什么都管的上帝对象。**纯函数式的约束是一种自律**——它强迫所有"需要看到全局对话"的逻辑上浮到能看到全局的那一层（Agent 循环层），而 LLM 层只保留"无状态就能做的事"。

这个认知直接决定了后面两个最重要的设计：**多 Provider 抽象**（管线内部换 Provider，上层零感知）和**重试职责的分层**（无状态重试留在管线内，有状态重试上浮到循环层）。

## 二、多 Provider 抽象：统一接口，对内分发

不同厂商的 API 长得完全不一样：OpenAI 的 `chat/completions`、Anthropic 的 Messages API、Gemini 的 `generateContent`，请求体结构、流式事件格式、usage 字段名全不同。

最朴素的做法是在调用处写 `if (provider === 'openai') {...} else if (...)`。这个做法的致命问题不是"丑"，而是**分发逻辑会泄漏到每一个调用点**——主对话要判断一次，摘要要判断一次，标题生成又要判断一次。每加一个 Provider，所有调用点都得改。

正确的抽象是：**对外暴露一个统一接口，对内按 Provider 类型分发。**

```
上层（loop / 摘要 / 画像 / 标题）
        ↓ 只调 streamChat / chatComplete
   streamChat（统一入口）
        ↓ detectProvider → 分发
  ┌─────┼─────┐
OpenAI  Anthropic  Gemini  （各自的请求构建 + SSE 解析）
```

关键判据：**Provider 的差异必须在管线内部被吸收干净，上层拿到的永远是同一种事件流。** 上层代码不知道、也不该知道底层是哪个 Provider。换 Provider、加 Provider，上层零修改。

这条抽象带来一个容易被忽视的**连带收益**：任何走了这条管线的调用，都自动获得全部 Provider 支持。反过来，任何绕过管线、自己拼请求的调用，就只支持它当初拼的那一家——这正是我们 M3 阶段最先要修的问题（见第二部分）。

## 三、非流式是流式的收敛，不是第二条路径

Agent 里有两类 LLM 调用：主对话要**流式**（用户要看到字一个个蹦出来），而辅助调用——生成标题、压缩摘要、提取画像——只要最终结果，不需要流式。

面对这个需求，直觉的做法是写两条路径：主对话走 `stream: true`，辅助调用走 `stream: false`，各自解析各自的响应。这个做法的代价是**你现在要维护两套请求组装 + 两套响应解析 + 两套错误处理 + 两套 usage 累积逻辑**。每次给管线加能力（新 Provider、cache、failover），都得改两个地方，还得记得两边对齐。

我们的选择是：**只保留流式一条路径，非流式是"把流式生成器消费到结束、取最终结果"的一层薄封装。**

```
chatComplete()  ——消费到底——>  streamChat()  ——流式——>  Provider
   （辅助调用）      取终态           （唯一管线）
```

`chatComplete` 内部就是一个 `while (!done) gen.next()` 循环，把事件流 drain 干净，返回累积好的完整文本。它复用了 streamChat 的**整条**能力——多 Provider、failover、Vision 降级、usage 解析——一行都不用重写。

这个设计我们后来在两个成熟框架（CC 的 `queryModelWithoutStreaming`、Alice 的单一流式接口）里都验证到了：**两家都不为非流式单独发请求，都是"单一流式管线 + 外层 drain"。** 甚至 CC 连"真正的 `stream: false` 请求"都只当作流式失败后的降级 fallback，而不是辅助调用的一等路径。

判据一句话：**输出形态的差异（流式 / 非流式）用"消费方式"区分，不用"请求路径"区分。** 请求永远只有一种。

## 四、重试分层：无状态重试下沉，有状态重试上浮

这是整个 LLM 层最微妙、也最值得讲清楚的一条。

Agent 会遇到好几种需要"重试"的失败，但它们不是一类东西：

| 失败类型 | 恢复动作 | 需要改对话内容吗？ |
|---------|---------|-----------------|
| 网络抖动 / 429 / 5xx | 等一会儿重发 | 否 |
| 主模型挂了 | 换个模型重发（failover） | 否 |
| 输出被 max_tokens 截断 | 调大输出上限重发 | 否（只改参数） |
| 输入超长 413 | **压缩对话历史**后重发 | **是** |

前三种有一个共同点：**重试时对话内容原封不动**，只是换个模型、调个参数、等一会儿。这类重试**不需要看到对话状态**，所以它们应该待在 LLM 层管线内部——`streamChat` 自己就能完成 failover，调用方甚至不知道发生过降级。

第四种是异类：413 的恢复动作是"把对话历史压缩变短"，这**必须重写 `messages`**。而重写 messages 需要访问完整的对话状态（`state.messages`）和压缩策略——这些东西 LLM 层根本不持有（回到第一节的纯函数约束）。所以 413 的压缩重试**必须上浮到能看到 state 的 Agent 循环层**。

于是判据就清晰了：

> **能在"不改对话内容"下完成的重试，下沉到 LLM 层；一旦需要重写 messages，就上浮到 Agent 循环层。**

这条线不是我们拍脑袋画的。M3 阶段我们一度怀疑"重试逻辑散在两层是不是重复了，要不要统一下沉到 LLM 层"，专门派 subagent 去审计 CC 怎么做的——结论是 CC 的分层跟我们**完全同构**，而且是刻意保留的（详见第二部分的 G4）。CC 甚至没有"让 LLM 层回调上层压缩"的钩子，它用的是"LLM 层把 413 降级成一条结构化数据、上层识别后自己压缩"——数据向上流，控制权始终在循环层。

**这条认知教会我们一件事：不是所有"看起来重复"的东西都该合并。** 两层各自有重试，是因为它们重试的是不同性质的失败。强行统一，反而会把纯函数层污染成有状态层。

## 五、usage 统计：流式累积的正确性陷阱

usage（token 消耗）看起来是个无关紧要的旁路统计，但它在流式解析里藏着一个隐蔽的正确性陷阱。

流式响应里，usage 不是一次性给全的，而是**散落在多个事件里、分批到达**。以 Anthropic 为例：

- `message_start` 事件先到，带 `input_tokens` 和 cache tokens（真实的输入统计）
- `message_delta` 事件后到，带最终的 `output_tokens`，但**不再带** input / cache

如果解析代码在收到 `message_delta` 时"重建"整个 usage 对象，就会把 `message_start` 设好的 input_tokens 和 cache tokens **全部丢掉**。这不是崩溃式的 bug，而是**统计悄悄算错了**——你的成本报表少算了输入和缓存，还找不到原因。

正确的处理是两条规则：

1. **合并更新，不重建**：后到的事件只更新它带来的字段（`output_tokens`），其余字段保留前面事件设好的值。
2. **`>0` guard**：只在拿到正数时才更新某个字段，防止某些代理在中间 chunk 塞 0 值覆盖掉真实统计。

判据：**流式累积的任何字段，都要假设"数据是分批、乱序、可能带 0 值到达的"，用合并 + guard 而不是覆盖。** 这条对 content 累积天然成立（大家都用 `+=`），但对 usage 这种"看起来该一次性拿到"的字段，最容易疏忽。

## 六、caller 归因：让每一次调用都能被追溯

一旦辅助调用也走了统一管线（第三节），会冒出一个新问题：日志里全是 `streamChat`，你分不清哪次是主对话、哪次是后台在压缩、哪次是画像提取在偷偷烧 token。

解法很轻：给管线入口加一个 `caller` 标识字段（`'main'` / `'summary'` / `'profile'` / `'title'`），调用方传进来，管线在日志里打出来。改动只有几行，但它把"一坨匿名调用"变成了"可归因的调用流"。

判据：**统一入口是好事，但统一之后要保留调用方的身份。** 归因信息是后续做 per-caller 成本统计、限流、优先级调度的地基——现在花几行埋下去，比事后从调用栈里反推便宜得多。

# 第二部分：实战记录

## M3 阶段做了什么

进入 M3 之前，我们对照 Alice 方法论和 CC 源码，给 LLM 层列了一串"gap"。深啃过程分两批落地、一个评估关闭：

**第一批（G1）——辅助调用统一收进路由层**

问题：摘要、画像提取、标题生成三个辅助功能，各自手拼 OpenAI 的 `chat/completions` 请求、各自 `fetch`、各自解析 `choices[0].message.content`。这意味着三份重复样板，而且**只支持 OpenAI**——用户一旦把模型切成 Claude 或 Gemini，这三个功能会静默失效（请求格式对不上）。

改动：新增 `chatComplete()` 非流式便捷入口（认知框架第三节的落地），三处调用全部改走它。连带收益立竿见影——三个辅助功能自动获得 Anthropic / Gemini 支持 + failover + Vision 降级，还删掉了三份 fetch 样板。同时埋了 `caller` 字段。

**第二批（G2 + G3）——两个正确性修复**

- **G2 usage guard**：Anthropic 的 `message_delta` 分支在重建 usage 对象时丢掉了 `message_start` 设好的 cache tokens（认知框架第五节的真实案例）。改成合并更新 + `>0` guard。
- **G3 retry-after**：`streamChat` 抛的是纯字符串 Error，把 429/503 响应里的 `retry-after` header 丢了，导致上层重试只能盲目指数退避。新增 `LLMError` 类携带 `status` + `retryAfterMs`，loop 重试时优先遵从服务端指定的等待时间。

**G5——caller 归因**：认知框架第六节的落地，几行改动。

每一步都补了单测，M3 期间单测从 106 增至 113。

## G4：一个决定"不填"的 gap

这是 M3 最有价值的一次判断，值得单独记。

我们原本列了 G4：**"重试职责下沉统一——把 loop 层的重试也搬到 LLM 层，消除两层都有重试逻辑的重复感。"** 听起来很合理：DRY 嘛，重复的东西就该合并。

动手前，我们先派 subagent 去审计 CC 是怎么处理的。结论让我们**放弃了 G4**：

1. CC 的 `queryModel` 也是纯函数、不持有对话状态——和我们的 `streamChat` 同构。
2. CC 的 413 输入压缩也在 Agent 循环层（`query.ts`），不在 LLM 层。
3. CC **没有**"LLM 层回调上层压缩"的钩子，用的是"LLM 层把 413 降级成结构化 error message、上层 withhold 识别后自己压缩"。
4. 连"单次尝试、失败即 surface"的死循环防护标志，CC 和我们都叫 `hasAttemptedReactiveCompact`（同名），思路完全一致。

也就是说：**我们以为的"gap"（两层都有重试），其实是一条正确的架构边界**（认知框架第四节）。把 413 压缩下沉到 LLM 层，等于让纯函数层去碰对话状态，破坏第一性原理；而"回调钩子"会让接口复杂化，CC 试都没试。

**教训**：审计参考实现的价值，不只是"抄它怎么做"，更是"确认我们没做的事是不是本来就不该做"。如果当初直接埋头做 G4，我们会花几天重构，把一个本来干净的分层搞脏，还自以为在消除重复。**先研究后协作，这次省下的是一次架构层面的返工。**

## 坑 1：mock 模块新增导出会漏

G3 给 `llm/index` 加了 `LLMError` 类并 export 后，`agent-loop.test.ts` 突然报错：`No "LLMError" export is defined on the mock`。原因是那个测试文件用 `vi.mock('../../electron/main/llm/index', ...)` **整体 mock** 了这个模块，只手动导出了 `streamChat`。loop 里新加的 `err instanceof LLMError` 引用到 mock 里不存在的符号，直接抛错。

解法：在 mock 工厂里补一个等价的 `LLMError` class（能用 `instanceof` 即可）。

**沉淀**：给一个被 `vi.mock` 整体 mock 的模块新增导出时，所有 mock 了它的测试文件都要同步补上，否则运行时才炸。这是"整体 mock"的固有脆弱性——它不会自动跟随真实模块的导出变化。

## 坑 2：先列 gap，再逐个质疑 gap

M3 一开始我们列了 G1-G5 五个 gap，默认它们都该做。实际结果是：G1/G2/G3/G5 做了，G4 评估后关闭。

差点犯的错误是**把"列出来的 gap"当成"待办清单"照单全收**。gap 是"当前实现和参考实现的差异"，但差异不等于缺陷——有些差异是我们更差，有些是我们恰好也对，还有些是参考实现的取舍不适用于我们。

**沉淀**：审计出的每个 gap，动手前都要再问一句"这个差异是真缺陷，还是只是不一样？"。填 gap 前先确认它值得填。

## 沉淀：LLM 层的设计检查清单

给未来加功能时用：

- [ ] 新的 LLM 调用有没有走统一管线（`streamChat` / `chatComplete`），还是又在手拼请求？
- [ ] 如果是"只要最终结果"的调用，用了 `chatComplete` 而不是新开一条非流式路径？
- [ ] 新增的重试逻辑：它需要改对话内容吗？需要 → 放循环层；不需要 → 放 LLM 层。
- [ ] 流式解析里新增的累积字段，用的是合并 + guard，还是覆盖？
- [ ] 抛出的错误有没有携带上层重试需要的信息（status / retry-after）？
- [ ] 新调用点传了 `caller` 吗？
- [ ] 给 `llm/index` 新增导出后，检查过所有 `vi.mock` 了它的测试文件吗？
- [ ] 这个"gap"动手前质疑过吗——它是真缺陷，还是只是和参考实现不一样？

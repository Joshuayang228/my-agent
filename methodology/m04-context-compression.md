# M4：上下文压缩工程化方法论

> 这份文档沉淀我们对 Agent 上下文压缩的设计思考。
> 前半部分是**认知框架**——压缩系统该怎么设计、为什么。
> 后半部分是**实战记录**——我们做了什么改动、踩了什么坑、哪些"gap"决定暂缓。

---

# 第一部分：认知框架

## 一、核心认知：上下文管理与记忆系统是两个问题

设计压缩系统前，先要认清一件容易混淆的事：**"当前对话放不下"和"上次对话怎么带过来"是两个独立的问题，解法完全不同。**

- **上下文管理**解决"这一次对话的信息超出窗口"——工具是压缩、截断、摘要。它是**短期**的、**会话内**的。
- **记忆系统**解决"跨会话把重要信息带过来"——工具是持久化存储、检索、注入。它是**长期**的、**跨会话**的。

很多团队踩的坑是把两者混为一谈：要么全用向量检索（结果连"当前对话太长"都靠召回，精准的结构化事实被语义检索得乱七八糟），要么全用摘要（跨会话记忆也被压成一段自由文本，越压越糊）。

M4 我们只做**上下文管理**这一半——压缩。记忆系统是 M5 的事。把边界划清楚，压缩系统才不会长出"我要不要顺便记住这个"的手。

## 二、分层压缩的本质：把"删什么"和"怎么删"分开

处理上下文溢出，最朴素的做法是"超了就摘要一次"——用一次 LLM 调用把长历史压成一段摘要。这能跑，但有个根本问题：**它对所有消息一视同仁，而消息的价值差异极大。**

第一条任务说明是整个任务的锚点，丢了它必然偏航；而重复读取同一文件、多次执行相同命令，它们的价值只在"最终结果"，早期的过程完全可以合并。用一套统一摘要处理这两类东西，质量必然不稳定——有时摘掉了关键意图，有时又花大篇幅保留无关的过程。

分层压缩的本质是：**把"什么该删"和"用什么方式删"这两个问题分开解决。**

我们的四层（对照 Alice Ch.5）：

| 层 | 触发 | 策略 | 成本 |
|----|------|------|------|
| L1 Snip | 60% | 删最早的工具调用轮次 | 零 |
| L2 MicroCompact | 75% | 去重相同工具调用 | 零 |
| L3 Collapse | 90% | LLM 摘要中段，保留首尾 | LLM 调用 |
| L4 AutoCompact | 95% | 全量重写 | LLM 调用 |

关键判据：**轻量的规则删除（去重、截断早期探索）不需要 LLM，成本为零；只有真正需要语义理解的压缩才调 LLM，而且调用时已经知道要压缩的是什么类型。** 阶梯式触发让早期轮次代价最小，重量级手段留到最后。

这个选择有代价——实现复杂、阈值要调。如果你的产品是短会话为主，简单摘要完全够用，不必上分层。加层的前提是你真的遇到了"某类信息用现有层处理不好"的问题。

## 三、preamble 是不可压缩的锚点

这是压缩系统里最容易出错、也最致命的一条边界。

所有压缩层都在"保留近期、压缩中段"。但有一段消息，无论多早、无论压缩多狠，都**绝不能碰**：**preamble——第一条 assistant 回复之前的所有消息（system prompt + 用户的初始任务说明）。**

原因很简单：任务说明是整个任务的意图锚点。它在对话最开头，正好落在"最该被压缩"的早期区间。如果压缩逻辑只保护 `messages[0]`（system prompt），用户那句"帮我重构认证模块"会在第一次 L3 collapse 时被摘进一句话，然后 AI 就开始偏航——它还记得在做事，但忘了为什么做。

判据：**保护边界不是"第一条消息"，而是"第一条 assistant 之前的整个 preamble"。** 这对齐 CC 的 `groupMessagesByApiRound`——第一条 assistant 开启 group 1，此前都是 group 0（preamble），永久保护。

实现上要把这条边界**统一到每一层**：L1 Snip、L3 Collapse、L4 AutoCompact 都用同一个 `getPreambleEndIndex`。任何一层漏了，任务说明就可能从那个缺口漏走。

## 四、压缩会破坏"AI 记得读过什么"，要主动补回

压缩有一个隐蔽的副作用：**它会让 AI 忘记自己读过的文件，然后重复去读。**

场景是这样的：AI 在中段用 `file_read` 读了 `auth.ts`，内容进了对话历史。随后 L3 collapse 把中段摘要成"读取了几个文件"，原始内容没了。下一轮 AI 想引用 `auth.ts` 的某个函数，发现上下文里只有一句摘要——于是它再 `file_read` 一次。压缩省下的 token，又被重复读取吃回去，还多绕一圈。

解法：**压缩后主动把最近读过的文件内容重新注入。** 但这里有个时序陷阱（见第五节），以及三个必须的约束：

1. **按路径去重**，保留最近一次读取——同一文件读了三次只恢复最新的。
2. **限量**——最多 5 个文件、单文件 5K token、总 50K token。恢复不能反过来撑爆上下文。
3. **错误结果不恢复**——`Error reading file` 这种没有恢复价值，跳过。

判据：**压缩不只是"删信息"，还要评估"删了之后 AI 会不会为了拿回信息而做重复劳动"。** 能预判的重复劳动（重读文件），在压缩时就把结果留住，比让 AI 事后重试便宜。

## 五、快照要在压缩链的最前面拍，不是用到时才拍

第四节的文件恢复，藏着一个"晚一步就拿不到"的时序陷阱，值得单独拎出来。

直觉的实现是：在 L3 collapse 里，从"即将被摘要的中段消息"里提取文件读取结果。**这是错的。** 因为 L1 Snip 在 L3 之前就跑了——它已经把最早的 `file_read` 轮次整轮删掉了。等 L3 想提取时，那些文件读取记录早没了。

正确做法：**在 `compressContext` 入口、任何压缩层运行之前，先对原始消息拍一次快照**，把文件读取状态捕获下来，再一路传给 L3/L4 使用。

```
compressContext 入口
   ├─ 拍快照（extractRecentFileReads）  ← 必须在这里
   ├─ L1 Snip（会删掉早期 file_read）
   ├─ L2 MicroCompact
   ├─ L3 Collapse（用快照恢复文件）
   └─ L4 AutoCompact（用快照恢复文件）
```

这对齐 CC 的 `preCompactReadFileState`——它也是在压缩前快照文件状态。

判据：**任何"压缩后要用到、但压缩过程中会被销毁"的信息，都必须在压缩链的最前面快照。** 用到时才去原地捞，那时东西可能已经被前面的层删了。这条不只适用于文件，适用于所有"压缩会破坏的中间态"。

## 六、递归防护：压缩自己不能触发压缩

L3/L4 会调用 LLM 生成摘要。但这个摘要调用**本身也是一次 LLM 调用**，如果不加防护，它会再次触发压缩检查，进入无限递归。同样的陷阱也存在于记忆提取、标题生成——它们都是 LLM 调用。

而且这个 bug 很难在测试时复现：单次测试几乎不触发，要同时有多个长任务或连续触发 AutoCompact 才稳定出现，症状还很怪（摘要错乱、消息序列被破坏）。

解法是**代码级的显式互斥守卫**：用一个 `querySource` 标记当前调用来源（`'main'` / `'compact'` / `'memory'` / `'title'`），非 `'main'` 的调用直接跳过 LLM 摘要（降级为规则占位符）。这个检查不能靠"感觉不会同时触发"，必须是显式判断。

判据：**任何"会调用 LLM 的压缩/记忆逻辑"，都要能识别自己是不是正在另一个压缩/记忆流程里，是则不再触发下一层。** 数据向上流（把状态降级成结构化标记），控制权不递归下探。

## 七、结构化摘要比自由文本更抗误读

L3/L4 调 LLM 做摘要时，摘要的**输出格式**是一个被低估的设计点。

自由文本摘要（"用户想重构认证模块，已经改了登录逻辑，还在处理登出……"）读起来没问题，但它在下一轮 LLM 推理时容易被误读——模型分不清哪句是"已完成"、哪句是"待办"、哪句是"关键约束"。

结构化框架把这些维度显式切开：

```
## 当前任务      [核心目标一句话]
## 已完成步骤    [有序列表]
## 当前状态      [进展 + 问题]
## 下一步计划    [待办]
## 关键上下文    [文件路径/变量名/配置值/用户偏好]
```

判据：**摘要是给下一轮 LLM 读的，不是给人读的——要按"LLM 最容易正确解读"的格式组织，而不是按"人读着顺"的格式。** 结构化约束还有个副作用：它降低了 LLM 摘要质量的波动性，因为模型被框架逼着逐项填，不容易漏掉某个维度。

## 八、熔断之后必须有降级，不能"什么都不做"

压缩会失败——LLM 摘要调用超时、返回空、或压缩后消息数没减少。连续失败时需要熔断器停止重试（否则每轮都白烧一次 LLM 调用）。但熔断之后呢？

朴素做法是"熔断后跳过压缩"。这留下一个致命缺口：**跳过压缩 ≠ 上下文变短**。下一轮 LLM 调用带着依然超长的上下文，直接 413 崩溃。熔断本是为了防止雪崩，结果自己制造了雪崩。

判据：**熔断后必须有一个"一定能成功、零成本、不失败"的降级动作兜底。** 我们用 `emergencyTruncate`——纯规则硬截断到目标 token（保护 preamble + 保留最近消息 + 移除孤儿 tool 消息）。它不调 LLM、不会失败，是最后的安全网。

对桌面伙伴产品，这里还有个产品判断：CC 在压缩彻底失败时抛 `ERROR_MESSAGE_PROMPT_TOO_LONG` 让用户手动处理。但我们的定位是"数字伙伴"，把 413 错误甩给用户不合适——宁可自动截断、丢一点早期上下文，也要让对话能继续。**同样的技术边界，产品定位不同，降级策略也不同。**

## 九、阈值要随模型窗口自适应

压缩触发阈值（60%/75%/90%/95%）是相对值，但"相对于什么"很关键。硬编码一个 `DEFAULT_MAX_TOKENS = 120K` 会在两个方向都出错：

- **大窗口模型过早压缩**：Gemini 有 1M 窗口，用 120K 当基准会在 72K（60%）就开始 Snip，白白浪费 90% 的可用空间。
- **小窗口模型压缩不及时**：若某模型只有 64K 窗口，用 120K 当基准，等你以为到 60% 时，实际早就超了真实窗口，直接 413。

解法：**按模型名推断真实 context window**，阈值基于"真实窗口 - 输出预留"计算。未知模型回退到保守默认值。

但这里有个更根本的判断：**只推断"跨代际稳定"的窗口，不猜会变的**。我们第一版把 GPT-4o(128K)/GPT-4.1(1M)/o1(200K)/DeepSeek(128K)/Qwen(128K) 全按前缀写死，结果两次翻车——DeepSeek 凭记忆写成 64K（实际 128K），Qwen 更是从 32K 到 10M 跨三个数量级无法用前缀区分。模型迭代比代码快，写死具体值等于埋定时炸弹。最终只保留 Claude(200K)、Gemini(1M) 两个多代稳定的家族，其余一律回退保守默认——**宁可略早压缩，也不赌一个会过时的数字**。真实窗口以 API 的 413 反压为准。

还有个兜底方向的细节：如果保留小窗口模型，`Math.max(window - reserve, floor)` 的 `floor` 不能取 `DEFAULT_MAX_TOKENS`——否则 56K 的真实窗口被兜成 120K，又回到"压缩不及时触发 413"。正确的下限是一个很小的安全值（16K），只防极端配置。

判据：**任何"相对于窗口大小"的阈值，都要拿到真实窗口再算；而"真实窗口"本身只在能长期稳定时才硬编码——会随版本变的参数，回退保守默认比猜一个具体值更安全。**

# 第二部分：实战记录

## M4 阶段做了什么

进入 M4 前，我们派三个 subagent 并行做"学 + 审"：一个提炼 Alice Ch.5 的压缩原则，一个深挖 CC `compact/` 源码，一个审计我们当前实现。产出 13 项 gap（P0 正确性 3 / P1 体验 6 / P2 优化 4），分三批落地。

**Phase A（正确性）——G1 + G4 + G11**

- **G1 保护任务说明**：新增 `getPreambleEndIndex`，统一 L1/L3/L4 保护 preamble（认知框架第三节）。
- **G4 压缩后文件恢复**：入口快照 + 恢复注入（第四、五节）。
- **G11 熔断降级**：`emergencyTruncate` 兜底（第八节）。

**Phase B（体验）——G3 + G12**

- **G3 结构化摘要**：L3/L4 摘要指令改结构化框架（第七节）。
- **G12 boundary marker**：`ChatMessage` 加 `compactMetadata`（层级/前后 token/触发源/是否用 LLM），压缩点可追溯。LLM 层序列化只取 role/content，元数据不泄漏 API。
- **G6（L4 独立会话）暂缓**：当前 `querySource` 防护已覆盖递归风险，独立 forked agent 实现成本高，收益边际——暂缓。

**Phase C（边界）——G7 + G10**

- **G7 PTL 逃生舱**：413 reactive compact 若没缩小消息，回退到 `emergencyTruncate` 硬截断再重试，而非直接放弃。
- **G10 动态阈值**：`getEffectiveContextWindow` 按模型推断窗口（第九节）。

单测 113 → 127（A×6 / B×3 / C×5）。全程 tsc 零错误。

## 坑 1：先列的 gap 隐含了错误的实现路径

Phase A 的 G4（文件恢复），我们最初的方案是"从被摘要的中段消息里提取 file_read 结果"。写完测试直接挂——因为 L1 Snip 早把那些轮次删了（认知框架第五节）。

这不是一个"写错了"的 bug，而是**方案本身建立在错误的时序假设上**。差点就顺着错误方案往下调，直到测试逼我们回头看数据流，才发现快照必须前移到入口。

**沉淀**：方案设计时就要在脑子里过一遍"数据流经过每一层后还剩什么"。尤其是"A 层产出、B 层消费"的跨层依赖，要确认中间的层不会把 A 的产出销毁。测试挂在这种地方，往往说明方案的时序假设错了，不是代码写错了。

## 坑 2：只保护 messages[0] 是个静默偏航源

同样在 Phase A，第一版 `collapse`/`autoCompact` 只保护 `messages[0]`（system）。测试构造"用户任务说明 → 大量工具轮次 → 触发压缩"后发现任务说明被摘掉了。

危险之处在于**它不报错**——压缩正常完成，消息数正常下降，一切看起来都对。只有 AI 的行为会慢慢偏航，而那时你已经很难把"跑偏"归因到"三十轮前的一次压缩吃掉了任务说明"。

**沉淀**：压缩这类"不会崩、只会让质量悄悄劣化"的逻辑，必须靠测试显式断言"关键信息还在"，不能靠"跑起来没报错"判断正确。我们专门加了 `TASK_MARKER` 断言测试。

## 坑 3：凭记忆写死会变的模型参数，翻了两次车

Phase C 的 G10，第一版把六七个模型的 context window 全按前缀写死。翻车两次：
1. DeepSeek 凭记忆写成 64K，实际是 128K——记忆里的数字过时了。
2. 想按代际给 GPT 写具体窗口（4o 128K / 4.1 1M），但 GPT/o 系列迭代太快、Qwen 更是从 32K 到 10M 跨三个数量级，按前缀根本区分不了。

用户一句"gpt 模型已经更新了很多代了，别写死"点破了本质：**这类参数的变化速度快于代码维护速度**。写死不是"写错一个值"，而是选错了策略——把一个高频变动的外部事实固化进代码。

**沉淀**：区分"该硬编码"和"该回退"的判据是**参数的变化频率 vs 代码维护频率**。Claude/Gemini 窗口多代稳定，可以写；GPT/DeepSeek/Qwen 会变或跨度大，回退保守默认 + 靠运行时反馈（API 的 413）兜底，比猜一个具体值更稳。另外，涉及外部事实（模型规格、价格、限额）不要凭记忆写——要么联网核实，要么干脆不写。

顺带一个兜底方向的细节：`Math.max(window - reserve, floor)` 的 `floor` 若取 `DEFAULT_MAX_TOKENS`，会把真实的小窗口兜高，反而制造"压缩不及时"。兜底要防的是"阈值被压到不可用"，取小值（16K）即可。

## 暂缓项：G6（L4 独立会话隔离）

Alice 和 CC 都用"独立 forked agent + 禁用工具"跑 L4 全量摘要，彻底隔离递归。我们评估后**暂缓**：当前的 `querySource` 互斥守卫已经防住了递归触发（摘要调用会被标记为 `'compact'`，不会再触发下一层压缩），残留风险是"摘要 LLM 理论上仍可能调工具"——但我们 L3/L4 的摘要调用走的是 `chatComplete`，本就不传 tools，这条路径实际上是关的。

判据：**gap 是"和参考实现的差异"，不等于"缺陷"。** G6 的差异在实现形式（forked agent vs querySource 守卫），不在防护效果——两者都堵住了递归。花大成本改实现形式换不到额外的安全，暂缓是对的。等未来真需要"摘要过程中调工具"（比如摘要时要读文件补全上下文），再上 forked agent 不迟。

## 沉淀：压缩系统的设计检查清单

给未来加功能/改压缩时用：

- [ ] 改的是"上下文管理"还是"记忆系统"？别把两者的逻辑混进同一个函数。
- [ ] 新增的压缩层，保护 preamble 了吗？用的是统一的 `getPreambleEndIndex` 吗？
- [ ] 这次压缩会不会让 AI 忘记什么、然后重复劳动？能预判的（读文件），压缩时留住结果了吗？
